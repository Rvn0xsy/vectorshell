# VectorShell Dev Doc

## Overview

VectorShell is an AI‑driven remote command execution platform written in Rust.

- **Server** hosts the AI agent, coordinates clients, and dispatches commands.
- **Client** connects via reverse WebSocket, executes commands, and returns results.
- **Shared** crate defines the JSON protocol types.

Core rules:
- Server controls all execution.
- Clients do not decide commands and contain no AI logic.
- All communication uses structured JSON.
- Clients are generated by the server and carry embedded config.

## Repository Layout

```
vectorshell/
├── server
│   └── src
│       ├── main.rs            # entry + REPL
│       ├── agent/             # AI command generator (rig)
│       ├── builder/           # client binary builder
│       ├── client_manager/    # tracks clients / routes exec
│       ├── config/            # config loader
│       └── websocket/         # WS server

│   └── src
│       ├── main.rs
│       ├── embedded_config/   # compile‑time config

│       └── websocket/         # WS client
├── shared
│   └── src
│       └── protocol.rs        # JSON messages
├── config
│   └── config.toml            # server config
└── docs
    └── DEV-DOC.md
```

## Build

```
cargo build
```

## Run (Local)

Terminal 1:
```
./target/debug/vectorshell-server --config config/config.toml
```

Terminal 2:
```
./target/debug/vectorshell-client
```

## Server REPL

Commands:
- `clients` — list connected clients
- `use <client_id>` — select a client
- `exec <command>` — execute on selected client
- `agent <task>` — ask AI to suggest a command (prints only)
- `agent-exec <task>` — AI suggests a command and dispatches it

## Configuration

`config.toml`

```
[server]
listen = "0.0.0.0:8080"
ws_path = "/ws"

[agent]
model = "gpt-4o"
base_url = "https://api.openai.com/v1"
api_key = ""   # optional; falls back to OPENAI_API_KEY
soul_path = "SOUL.md"

[client]
default_server = "wss://127.0.0.1:8080/ws"
reconnect_interval = 5
insecure_tls = false

[auth]
api_token = "change-me-api-token"
client_token = "change-me-client-token"
```

The server loads agent instructions from `SOUL.md` through `agent.soul_path`.

## Client Generation

Build a client binary directly:

```
go build -o build/clients/vectorshell-client ./cmd/client
```

## Embedded Client Config

Client embedded overrides live in `internal/embedded`.
They are injected at build time via `-ldflags` when needed.

## Protocol

Base message format:

```
{ "type": "string", "id": "uuid", "payload": {} }
```

Core message types:

Client → Server:
- `register`
- `heartbeat`
- `result`

Server → Client:
- `exec`
- `upload` (placeholder)
- `download` (placeholder)
- `ping` (placeholder)

## AI Agent

The Go server agent loads its instruction text from the file configured by `agent.soul_path`.

- Supports `base_url` for OpenAI-compatible APIs.
- `api_key` can be configured in TOML or via `OPENAI_API_KEY`.

## Logging

Server logs:
- client connections
- command dispatch
- command results
- config load events

Uses `tracing` + `tracing-subscriber`.

## Notes for Contributors

- Keep changes minimal and avoid unnecessary abstractions.
- Maintain JSON‑structured communication.
- Do not add AI logic to the client.

## Code Style & Module Conventions

- Prefer small modules with single responsibility.
- Keep shared protocol types in `shared/src/protocol.rs` only.
- Server modules live in `server/src/*` and should avoid cross‑module cyclic deps.
- Client modules live in `client/src/*` and must remain AI‑free.
- Avoid global state; use explicit structs and state holders.
- Use `tracing` for server logs; avoid `println!` except for REPL output.
- Keep protocol JSON stable; changes must update both server/client.

## Test & Verification Flow

Minimum local checks before PR or release:

```
cargo build
```

Optional smoke test (manual):

1. Start server: `vectorshell-server --config config/config.toml`
2. Start client: `vectorshell-client`
3. In server REPL:
   - `clients`
   - `use <client_id>`
   - `exec whoami`
   - (optional) `agent <task>` / `agent-exec <task>` if API key set

Expected behavior:
- Client registers successfully
- `exec` returns `result` with `exit_code=0`
